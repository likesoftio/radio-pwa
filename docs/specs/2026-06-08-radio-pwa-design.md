# Дизайн: PWA «Радио»

**Дата:** 2026-06-08
**Статус:** утверждён, готов к написанию плана реализации
**Подпроект:** `ai-agency/radio-pwa/`

## Вводные (зафиксированные решения)

- **Суть:** мобильное онлайн-радио в виде устанавливаемого PWA. Каталог станций для русскоязычной аудитории и станций со всего мира.
- **Источник станций:** открытый API [radio-browser.info](https://api.radio-browser.info) — каталог потоков, поиск, фильтры, метаданные. Своего каталога нет.
- **Бэкенд:** отсутствует. Полностью клиентское приложение; браузер ходит в API и в потоки напрямую.
- **Данные пользователя:** только локально (IndexedDB через Dexie). Без регистрации и аккаунтов.
- **Язык интерфейса:** только русский. Строки вынесены в словарь, чтобы i18n можно было добавить позже без переписывания.
- **Стек:** React + TypeScript + Vite, `vite-plugin-pwa` (Workbox), Zustand (состояние), Dexie.js (IndexedDB).

## Объём MVP

1. **Поиск и фильтры** — поиск по названию, фильтры по стране / жанру / языку.
2. **Избранное** — отметить станцию и быстрый доступ к отмеченным.
3. **Фоновое воспроизведение** — играет при заблокированном экране, управление через Media Session API.
4. **Название трека** — показывать текущий трек, если станция отдаёт ICY-метаданные (с честным фолбэком на название станции).

## 1. Архитектура (высокий уровень)

Полностью клиентский SPA + PWA. Нет своего сервера. Браузер обращается напрямую к:
- **radio-browser.info API** — каталог станций (поиск, фильтры, метаданные);
- **потоку станции** — напрямую в элемент `<audio>`.

```
[React UI] → [radio-browser API]  (каталог)
     │
     ├─→ [<audio> элемент] → [HTTP-поток станции]   (звук)
     ├─→ [Media Session API]                         (фон / экран блокировки)
     └─→ [Dexie/IndexedDB]                           (избранное, история, настройки)
```

## 2. Структура каталога

Новый подпроект `ai-agency/radio-pwa/` со своим git-репозиторием и тулчейном.

```
radio-pwa/
  index.html
  vite.config.ts          # + vite-plugin-pwa (manifest, SW, offline-shell)
  src/
    main.tsx
    app/App.tsx           # роутинг (3 экрана), плеер всегда внизу
    features/
      catalog/            # поиск, фильтры, список станций
      favorites/          # избранное
      player/             # аудио-движок + Media Session + ICY-метаданные
    shared/
      api/radioBrowser.ts # клиент API + выбор рабочего зеркала
      db/db.ts            # Dexie-схема (favorites, history, settings)
      ui/                 # переиспользуемые компоненты
    i18n/ru.ts            # все строки в словаре (на будущее i18n)
```

## 3. Ключевые компоненты

Каждая единица имеет одну ответственность, общается через явный интерфейс и тестируется изолированно.

- **`shared/api/radioBrowser.ts`** — клиент API. Сам определяет живое зеркало (`all.api.radio-browser.info` отдаёт список серверов), методы `search()`, `topClick()`, `byCountry()`, `byTag()`, `byLanguage()`. Зависит только от `fetch`.
- **`features/player` (Zustand store)** — единый аудио-движок: одна `<audio>` на всё приложение, состояние `play/pause/buffering/error`, текущая станция. Управляет Media Session (кнопки на экране блокировки) и читает ICY-метаданные названия трека.
- **`shared/db/db.ts` (Dexie)** — таблицы `favorites`, `history`, `settings`. Простые async-методы; UI не знает про детали IndexedDB.
- **`features/catalog`** — поиск с debounce, фильтры (страна / жанр / язык), бесконечная прокрутка списка.
- **`features/favorites`** — читает из db, отметить/снять одним тапом, реактивное обновление списка.

## 4. Поток данных

1. Старт → выбор рабочего зеркала API → загрузка топовых станций.
2. Тап по станции → `player.play(station)` → `<audio>.src = url`, обновляется Media Session.
3. Звезда → запись в Dexie → экран «Избранное» реактивно обновляется.
4. ICY-метаданные: основной путь — поле `now playing` из потока; если CORS не даёт прочитать, показываем название станции (честный фолбэк).

## 5. Обработка ошибок

- Мёртвое зеркало API → перебор следующего из списка серверов.
- Поток не открывается / обрыв → статус «Не удалось подключиться», кнопка «Повторить»; станция не «залипает» в состоянии play.
- Офлайн → app-shell открывается из кэша Service Worker; список станций показывает «Нет сети». Потоки всё равно требуют интернет — это честно сообщается.

## 6. PWA-специфика

- `manifest.webmanifest`: имя, иконки (192/512 + maskable), `display: standalone`, тема.
- Service Worker (Workbox через `vite-plugin-pwa`): кэш app-shell и статики для offline-запуска. Аудио-потоки **не** кэшируются.
- Кнопка «Установить» через событие `beforeinstallprompt`.

## 7. Тестирование

- **Unit (Vitest):** `radioBrowser` (выбор зеркала, парсинг ответов), `db` (CRUD избранного), логика player-стора.
- **Компонентные (Testing Library):** карточка станции, панель фильтров, тоггл избранного.
- Сетевые вызовы мокаются; реальные потоки в авто-тестах не дёргаются.

## 8. Сознательно вне MVP (YAGNI)

Аккаунты и синхронизация между устройствами, рекомендации, эквалайзер, таймер сна, шеринг, мультиязычность интерфейса (строки готовы к i18n, но активен один язык). Добавляется позже при необходимости.
